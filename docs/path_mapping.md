# リンク置換（URL・パスマッピング）機能仕様書

## 概要
DNSが存在しない社内ネットワーク環境において、サーバーの移行やリプレース（約5年周期など）が行われると、IPアドレスやホスト名が変更され、過去の資料やObsidianのMarkdownドキュメント等に記載された古いリンク（UNCパスやWeb URL）がリンク切れを起こします。

本機能は、古いアドレスから新しいアドレスへの変換定義を保持し、アプリケーションがファイルをオープンする際（またはAPI経由でアクセスされた際）に自動的にリンクを最新の有効なものへとリライトし、アクセスをシームレスに継続できるようにするものです。

---

## 1. 設定仕様

設定ファイル `settings.json` の中に `pathMappings` キーとして辞書形式で定義します。
50個以上の大量のマッピングが登録される状況や、移行時期ごとに一括バックアップ・コピペしやすいよう、JSON文字列での編集をベースとします。

### データ構造
- **キー (左側)**: 新しい（生存している）サーバーのアドレス/プレフィックス
- **値 (右側)**: 古い（移行前の）サーバーのアドレス/プレフィックス（カンマ `,` で区切ることで複数指定可能）

### JSON入力例
```json
{
  "textFileOpenMode": "web",
  "markdownOpenMode": "web",
  "apiTimeout": 10,
  "pathMappings": {
    "\\\\new-server\\share": "\\\\old-1\\share,\\\\old-2\\share",
    "http://new-wiki/": "http://old-wiki-1/,http://old-wiki-2/"
  }
}
```

> [!NOTE]
> サーバー名やホスト名、IPアドレス、URLのドメインスキームには仕様上カンマ `,` は一切使用されないため、カンマでの複数旧サーバー指定は競合なく安全に行うことができます。

---

## 2. 変換処理ロジック (バックエンド)

バックエンドの `app/routers/files.py` の `convert_storage_path(path: str) -> str` で変換を行います。
この処理は `normalize_path` の前、および `open_smart` の冒頭部で呼び出されます。

### 特徴
- **URLとパスの自動識別**:
  - `http://`, `https://`, `obsidian://` から始まる入力はURLとして扱い、文字列前方一致で置換します。
  - ローカルパスやWindowsのUNCパスの場合は、区切り文字（`/` と `\`）の違いや大文字小文字の違いを無視して判定します。
- **カンマ区切りの自動フラット化**:
  - 設定ロード時、キー（新）と値（旧、カンマ区切り）を展開し、内部で逆引き用の 1対1 マッピング（`{旧: 新}`）に平坦化して探索します。
- **区切り文字の統一**:
  - パス置換後の結合時に、重複したスラッシュなどを排除し、入力パスに含まれていた元々の区切り文字（`\` が含まれていれば `\`）に統一して再構築します。これにより、OS毎のUNCパス構造が破壊されるのを防ぎます。

---

## 3. スマートオープン (Bypass)

古いWeb URLリンク（例: `http://old-server/page`）が `open_smart` に投げられ、新しいURL（例: `http://new-server/page`）に置換された場合、ファイルシステムのパスではないため、そのままパスの存在検証 `Path.exists()` や正規化 `normalize_path()` に送るとエラーになります。

そのため、変換後のアドレスが `http://`, `https://`, `obsidian://` などのURLスキームであると判定された場合、**即座に処理をバイパスし、webbrowserやOS既定のコマンド（`open` / `start` 等）を使用して直接URLを開く早期リターン**を実施します。

---

## 4. フロントエンド UI (編集モーダル)

1. **アクセス方法**:
   - 右上のハンバーガーメニュー ➔ **「リンク置換設定の編集...」** をクリックします。
2. **モーダルの構成 (ハイブリッドUI)**:
   - **入力フォームで編集（簡易表示、推奨）**:
     - 直感的で人間に優しいリスト形式の画面です。
     - **左側インプットボックス**: 移行先の「新サーバー」（例: `\\new-server\share`）を入力します。
     - **右側インプットボックス**: リンク切れしている「旧サーバー群」（複数ある場合はカンマ `,` で区切る、例: `\\old-1\share,\\old-2\share`）を入力します。
     - 各行の右端にある「ゴミ箱アイコン」ボタンでルールを削除でき、最下部の「＋ ルールを追加」ボタンでいつでも新規ルールを挿入できます。
   - **JSONで直接編集（詳細表示）**:
     - 設定データ全体をインデント付きのJSON形式の `textarea` で一括編集・コピペ可能にします。
     - 大量のデータ移行や設定のバックアップを行いたい場合に適しています。
3. **同期とバリデーション**:
   - モードを切り替えるトグルボタンにより、フォームの入力とJSONデータは双方向にリアルタイムで自動パース・同期されます。
   - 不正なJSON形式が入力された状態で「入力フォームで編集」タブや「保存」を押した場合は、赤字でパースエラー原因（JSONSyntaxError等）を警告表示し、誤ったデータの保存を防ぎます。

---

## 5. 新サーバーの接続テスト（疎通確認）

リンク置換設定（マッピングルール）の誤りを防ぐため、入力された「新サーバーのアドレス」が実際にアクセス可能であるかを検証するテスト機能を提供しています。

### UI操作と機能
- **「新サーバーの接続確認」ボタン**:
  - 設定編集モーダルの右上（タブ切り替え行の右端）に配置されています。
  - クリックすると、現在入力されている「新サーバー」のアドレス一覧がバックエンドの検証API（`POST /api/config/test-connections`）へ送信されます。
  - JSON直接編集モード時にボタンを押した場合は、JSONの内容を自動でパースしてフォーム形式に切り替えた上でテストを開始します。
- **検証ステータス表示**:
  - テストが開始されると、各入力行の左端にインジケータが表示されます：
    - **ローディング（青色スピナー 🔄）**: 接続確認の通信中。
    - **チェックマーク（緑色 ✅）**: 接続成功（Web URLの場合はHTTPステータスが200〜300系、パスの場合はファイル/フォルダの実在を確認）。
    - **警告マーク（赤色 ⚠️）**: 接続失敗。アイコンにマウスカーソルを合わせることで、失敗理由（「パスが見つかりません」「Connection Timeout」など）がツールチップで表示されます。

### バックエンド検証の仕様
- **URL (HTTP/HTTPS)**: 3秒の接続タイムアウト付きでHEADリクエストを送信し、200〜399の応答ステータスを確認します。
- **ファイル/UNCパス**: スレッドプール内で安全に実在検証を実行し、ネットワーク共有への応答がない場合のアプリケーションハングを防ぎます。
- **カスタムURI (obsidian://等)**: 形式が合致している場合は検証をパス（True）として扱います。
